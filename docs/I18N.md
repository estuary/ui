# i18n

## Setup

The app uses `react-intl`. The `IntlProvider` is configured in `src/context/Content.tsx` (the outermost provider in `src/context/index.tsx`).

```typescript
<IntlProvider
    defaultLocale="en-US"
    locale={navigator.language || 'en-US'}
    messages={enUSMessages}
    onError={(err) => {
        // Missing keys are logged to LogRocket
        logRocketEvent(CustomEvents.TRANSLATION_KEY_MISSING, { key: err?.descriptor?.id });
    }}
>
```

**Only `en-US` is currently implemented.** There is a TODO in `src/context/Content.tsx` to not hardcode English. No language-switching UI exists.

## Message Files

All messages live in `src/lang/en-US/`, split by domain:

```
src/lang/en-US/
  index.ts           - aggregates all files into one Record<string, string>
  AccessGrants.ts
  AdminPage.ts
  Alerts.ts
  Authentication.ts
  Captures.ts
  Collections.ts
  CommonMessages.ts  - shared across features
  Connectors.ts
  CTAs.ts
  Details.ts
  EntityTable.ts
  Errors.ts
  Workflows.ts
```

The `index.ts` spreads all objects into a single flat map. There is no nesting at runtime — all keys must be globally unique.

## Usage in Components

### `useIntl()` hook

Use when the translated string needs to be passed as a prop or used in logic:

If you need to share the content

```typescript
import { useIntl } from 'react-intl';

const intl = useIntl();
const label = intl.formatMessage({ id: 'common.loading' });
```

If you are using it in a single location

```typescript
import { useIntl } from 'react-intl';

const intl = useIntl();


<Typography>
    {intl.formatMessage({ id: 'common.loading' })}
</Typography>
```

### `<FormattedMessage>` component

Do NOT use this approach - it is being phased out. You should always use the `useIntl()` hook approach

```typescript
import { FormattedMessage } from 'react-intl';

<Typography>
    <FormattedMessage id="error.title" />
</Typography>
```

### Message IDs with variables

If you want to pass data into the message you can pass variables in with the function call.

```typescript
intl.formatMessage({ id: 'capture.details.count' }, { count: bindings.length });
```

## Naming Conventions

IDs follow a dot-separated hierarchy: `domain.section.key`

This was initially done in a way where the IDs were based on the name of the `.TSX` files they were consumed in. This matching is no longer always done.

Examples from the codebase:

```
error.title
error.reason.fetchFailed
error.messageLabel
common.loading
common.running
common.errors.heading
workflows.error.initForm
workflows.error.endpointConfig.empty
workflows.collectionSelector.cta.rediscover
terms.connectors
terms.bindings
routeTitle.captures
routeTitle.captureCreate
custom.prefixedName.noAccessGrants
custom.catalogName.pattern
alert.error
alert.warning
```

**Rules:**

- Lowercase, dot-separated
- Domain first (`error`, `common`, `workflows`, `terms`, `routeTitle`, `custom`)
- Specific enough to be searchable, not so specific they duplicate meaning

## Adding a New Message

1. Search existing messages and see if it is truly new
1. Find the appropriate file in `src/lang/en-US/` (or create one if the domain is new)
1. Add the key-value pair to the exported object
1. If creating a new file, import and spread it in `src/lang/en-US/index.ts`
1. Use the ID in components via `useIntl()` or `<FormattedMessage>`

All keys must be unique across the entire `src/lang/en-US/` directory since they are merged into a flat map at runtime.
